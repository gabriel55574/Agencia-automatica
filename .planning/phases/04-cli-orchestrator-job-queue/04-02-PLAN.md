<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Supabase Realtime | Client component subscribes to squad_jobs UPDATE events |
| Supabase Realtime → component state | payload.new data set directly as SquadJob state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-09 | Information Disclosure | progress_log displayed in modal | accept | progress_log contains Claude CLI stdout — squad output for this operator's own jobs. Solo operator context: no multi-tenant concern, no other user can access this modal. RLS on squad_jobs is not enforced at Realtime level (service role key bypasses RLS), but Supabase Realtime for browser clients uses the anon key with RLS, so other users cannot subscribe to another client's job events. Acceptable for v1. |
| T-04-10 | Tampering | payload.new cast as SquadJob without validation | mitigate | Validate Realtime payload with squadJobSchema.safeParse(payload.new) before calling setJob(). If parse fails, log the error and ignore the update. This prevents a malformed Realtime payload from corrupting component state. |
| T-04-11 | Denial of Service | Uncontrolled Realtime subscriptions (many open modals) | accept | Solo operator, single browser session. At most one progress modal open at a time. Cleanup via removeChannel() in useEffect return handles subscription teardown. Acceptable for v1. |
</threat_model>
