## Translation Strategy

### Approach: Batch by Functional Area
Translate in logical groups to maintain consistency within each area:
1. **Wave 1 — Core layout & navigation** (layout, sidebar, app page headers) — these set the overall language tone
2. **Wave 2 — Client management** (client form, grid, card, dialogs) — highest-traffic area
3. **Wave 3 — Pipeline & gates** (pipeline-accordion, pipeline-phase, process-row, gate-section, gate-review-display) — core domain terminology
4. **Wave 4 — Dashboard** (Kanban, cost widget, action panel, bottleneck alert) — main operational view
5. **Wave 5 — Documents, squad, costs, analytics** — supporting views
6. **Wave 6 — Server actions & email templates** — user-facing messages
7. **Wave 7 — Verification sweep** — final grep to catch any missed strings

## Technical Considerations

1. **No i18n framework**: All translations are inline hardcoded per D-05. Each component file is self-contained.
2. **String interpolation**: Some strings include dynamic values (`{clientName}`, `{gate.gate_number}`). Use Portuguese template literals that preserve the variable.
3. **Pluralization**: Portuguese requires gender agreement. "Nenhum cliente" (masculine), "Nenhuma execucao" (feminine). Apply correct grammatical gender.
4. **Action error messages**: These are returned from server actions and displayed via toast or form validation. Translate to PT-BR for consistency.
5. **Zod validation messages**: Default Zod error messages are in English. Custom messages in the action files should be translated. Built-in Zod messages (e.g., "Required") are not user-facing in this app (custom messages override them).

## Risk Assessment

- **LOW RISK**: This is a text-replacement phase. No logic changes, no architectural changes, no new components.
- **MEDIUM RISK**: Missing a string in a less-visible component (empty state, error path). Mitigated by the verification sweep (Wave 7).
- **NO RISK to LOC-02**: PHASE_NAMES already in PT-BR in `enums.ts`. Just verify they render correctly.
