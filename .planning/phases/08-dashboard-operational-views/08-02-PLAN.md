**1. Create `src/hooks/useRealtimeDashboard.ts`:**

Per D-09, D-10, D-11, D-12: Single Supabase Realtime channel subscribing to clients, squad_jobs, and quality_gates tables.

```typescript
'use client'
import { useEffect, useState, useCallback } from 'react'
import { createClient } from '@/lib/supabase/client'
import type { DashboardData } from '@/lib/dashboard/types'
```

Hook signature: `export function useRealtimeDashboard(initialData: DashboardData): DashboardData`

Implementation:
- `const [data, setData] = useState<DashboardData>(initialData)`
- `useEffect` that syncs `initialData` into state when it changes (for searchParam toggles): `setData(initialData)` in a `useEffect` with `[initialData]` dep
- `useEffect` for Realtime subscription:
  - Create browser Supabase client: `const supabase = createClient()`
  - Create a single channel: `const channel = supabase.channel('dashboard-realtime')`
  - Per D-12: single channel with multiple table filters
  - Subscribe to three tables:
    ```
    channel
      .on('postgres_changes', { event: '*', schema: 'public', table: 'clients' }, handleClientChange)
      .on('postgres_changes', { event: '*', schema: 'public', table: 'squad_jobs' }, handleJobChange)
      .on('postgres_changes', { event: '*', schema: 'public', table: 'quality_gates' }, handleGateChange)
      .subscribe()
    ```
  - Implement the pragmatic pattern: on ANY postgres_changes event from any of the three tables, call `router.refresh()` to re-fetch server data. This avoids complex client-side state reconciliation while still providing real-time updates. The UX is: Realtime notifies the browser, browser triggers a soft refresh, RSC re-fetches with full data joins.
  - To avoid spamming refreshes (multiple rapid changes), debounce the refresh with a 500ms window using a simple timeout pattern:
    ```
    let refreshTimer: NodeJS.Timeout | null = null
    const debouncedRefresh = () => {
      if (refreshTimer) clearTimeout(refreshTimer)
      refreshTimer = setTimeout(() => router.refresh(), 500)
    }
    ```
  - All three handlers (`handleClientChange`, `handleJobChange`, `handleGateChange`) simply call `debouncedRefresh()`
  - Cleanup: `return () => { supabase.removeChannel(channel); if (refreshTimer) clearTimeout(refreshTimer) }`
- Return `data`

Import `useRouter` from `next/navigation`.

The hook returns the initial data (which gets refreshed by RSC after router.refresh()). The `initialData` prop changes on each RSC re-render, which the useEffect syncs into local state.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client browser -> Supabase Realtime | Browser subscribes to table changes; RLS enforces access |
| Realtime payload -> state update | Payload triggers router.refresh(), not direct state mutation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-04 | I (Info Disclosure) | Realtime subscription | mitigate | RLS on all tables (Phase 1) restricts Realtime events to authenticated users. Supabase Realtime respects RLS policies by default. |
| T-08-05 | T (Tampering) | handleClientChange | mitigate | Realtime payloads are not used to directly mutate state. They only trigger router.refresh() which re-fetches from server with full auth. No client-side data trust. |
| T-08-06 | D (DoS) | Realtime debounce | mitigate | 500ms debounce prevents rapid-fire refreshes from many concurrent changes. Single channel for all three tables reduces connection overhead. |
</threat_model>
